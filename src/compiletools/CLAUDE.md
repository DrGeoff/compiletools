## Locking system: full invariant rationale

Two invariants for concurrent peer-`make` safety on the object CAS. **Violations produce sporadic `undefined reference to 'main'` from `crt1.o:_start`** — most easily reproduced under PyPy, where slower interpreter timing widens the window.

**1. Lock a sidecar, never the build target.** All four strategies lock `<target>.lock` (or `<target>.lockdir` for `LockdirLock`). Locking the target directly via `os.open(target, O_CREAT|O_RDWR)` creates an empty target file with `mtime=now` *before* the inner compile runs. A concurrent peer `make` then statting the target sees `mtime(target) > mtime(source)` and treats it as up-to-date — skipping the compile recipe entirely and linking an empty `.o`. The sidecar is invisible to make's dependency graph.

**2. Producer-side temp+rename.** The compiler writes to `<target>.compiletools.tmp` and `mv -f`s into place. Link rules read `.o` files without acquiring any lock — read-side locks would either explode the process count or invert the lock-acquisition order and deadlock. The rename guarantees a peer linker mmap-reading the target sees either the previous good `.o` (old inode) or the new one (new inode), never partial bytes.

Both invariants must be maintained in every code path that emits compile/link commands: the native-`flock` fastpaths in `wrap_compile_with_lock`/`wrap_link_with_lock` (Make and Ninja makefiles), and the in-Python `atomic_compile`/`atomic_link` callers (Shake/Slurm via `trace_backend.py`). `trim_cache._safe_locked_rmtree` filters `*.lock`/`*.lock.excl` from both the lock list and its TOCTOU re-scan — its own sidecars would otherwise be misread as peer build activity.

**Signal-window invariant: protection is installed BEFORE the resource it covers exists.** A signal handler that unwinds the stack (`sys.exit`, re-raise) must never be the active handler while a subprocess exists outside a kill-on-exit scope, and never act on a resource registry that lags the resource. Three enforcement points: (1) `_forwarded_session_child` (`locking.py`) installs the SIGINT/SIGTERM forwarder *before* `Popen` and latches signals arriving mid-spawn (`holder["pending"]`), forwarding them right after the child handle exists — `_run_with_signal_forwarding` must spawn only through it (regression: `test_sigterm_in_spawn_window_reaps_child`, which widens the window via a slow-Popen shim). (2) `SlurmBackend._tracked_submit` (`trace_backend.py`) latches signals across the sbatch submit → `_tracked_jobs` insert so `_scancel_pending` never runs while a submitted array id is untracked (regression: `TestSignalDuringSubmit`). (3) `ct_lock_helper.GracefulExit.cleanup` releases NOTHING directly — while blocked in `lock.acquire()` the lock belongs to a peer, and `LockdirLock`/`CIFSLock.release()` remove peer-visible state unconditionally; owned-resource cleanup rides the SystemExit unwind through `_temp_under_lock` (temp-before-release ordering) and `_forwarded_session_child` (regression: `TestSigtermWhileWaitingForContendedLock`).

## PCH consumer wiring: why `-include`, not `-I`

Consumer wiring is `-include <cas-pchdir>/<hash>/<basename>`, NOT `-I <cas-pchdir>/<hash>`. GCC's `#include "header"` resolution searches the source-file directory before any `-I` directory, so an `-I <cache>` flag is silently bypassed whenever the PCH header coexists with the consumer source — the realistic case for a private per-TU PCH (header and consumer `.cpp` co-located in the same directory). Source-dir resolution finds the bare `.h`, GCC then looks for a sibling `.gch` only beside that resolved path, never reaching the cache. The absolute `-include` form opens `<cache>/<basename>.gch` unconditionally; the consumer's own `#include "header"` is a no-op via the include guard.

Anyone tempted to "simplify" the emitted flag back to `-I` will silently revert this fix. Regression guard: `test_pch_bypass_bug.py`, parameterized over all six backends (make, ninja, cmake, bazel, shake, slurm) — asserts the consumer compile's `-H` output contains a `! …gch` PCH-used marker (`x …gch` reject markers also count as evidence the cache was reached, distinguishing the bypass bug from a separate flag-mismatch concern).

`_create_compile_rule` calls `_stage_pch_header_alongside_gch` to hardlink (with EXDEV-tolerant copy fallback) the source `.h` next to the `.gch`. Without it, gcc errors `No such file or directory` whenever the cache is invalidated at consume time and gcc tries to fall back to the bare header.

## Bazel PCH staging + dialect alignment

Bazel's hermetic `CcCompileAction` rejects absolute paths outside the toolchain's `cxx_builtin_include_directories`, and `additional_compiler_inputs` only accepts workspace-relative labels — so the cas-pchdir absolute path that `_create_compile_rule` emits cannot reach a bazel sandbox directly. `BazelBackend._bazel_pch_inputs_and_copts` (planning) records the cas → workspace mapping and rewrites the `-include` flag to `<workspace>/.ct-bazel-pch/<hash>/<basename>`; `_materialise_pch_stagings` (called from `_execute_build` AFTER `_prebuild_aux_artefacts` builds the .gch) hardlinks both the `.h` and the sibling `.gch` into `.ct-bazel-pch/`. Bazel symlinks each declared input into the action's exec root at the same workspace-relative path; gcc's PCH lookup finds the sibling `.gch` and validates.

For PCH validation to actually succeed, the dialect bazel uses for the consumer compile must match the dialect ct-cake used for the PCH. Bazel's `local_config_cc` autoconfig defaults to `-std=c++17` (strict ISO), which undefines built-ins like `unix`, `linux`, `__unix__` that ct-cake's gcc-default `-std=gnu++NN` mode kept defined, producing `PCH not used because 'unix' not defined` rejections. `_build_bazelrc_content` therefore unconditionally emits `build --cxxopt=-std=<X>` where `<X>` is the user's explicit `-std=` if set in CXXFLAGS, otherwise `apptools.compiler_default_cxx_std(args.CXX)` — a per-compiler probe (`<cxx> -dM -E -x c++ /dev/null`, parses `__cplusplus`, maps to `gnu++NN` — gcc-16 → gnu++20, clang-21 → gnu++17). The `gnu++` mode is the explicit choice: both gcc and clang default to gnu mode, and switching to strict `c++` would itself invalidate any prebuilt PCH that recorded gnu-mode built-ins. Regression guards: `test_pch_bypass_bug.py::test_consumer_compile_actually_opens_cached_pch[bazel]` and `test_apptools.py::TestCompilerDefaultCxxStd`.

## C++20 modules: discovery and per-compiler details

**Discovery flow.** `file_analyzer.analyze_file()` scans every source for module declarations (`export module M(:P);`, `module M(:P);`, `import M(:P);`, `import :P;`, `import <h>;`, `import "h";`). Hunter builds a `module_name → first-exporter-path` registry from the global hash registry (with an `os.walk` fallback for non-git tmp dirs and any explicit `--include` directories). Multiple files exporting the same name are tolerated at registry-build time (warning at `verbose ≥ 1`); the hard error fires at lookup time with the importer's path in the diagnostic. System modules (`std`) fall back to compiler-shipped sources: `<gcc-include>/c++/<ver>/bits/std.cc` for gcc, `<clang-install>/share/libc++/v1/std.cppm` for clang.

**Why single command_hash + manifest, not the object cache's 3-axis path?** The object cache uses `{basename}_{file_hash_12}_{dep_hash_14}_{macro_state_hash_16}.o` (~168 bits in the path) because there is **no in-band verification** of `.o` content at link time — the linker happily links the bytes it's handed. A hash collision would cause a silent miscompile, so the path needs the entropy of multiple independent hashes. PCH and PCM are different: **the compiler verifies BMI compatibility at consume time** (GCC's PCH stamp, clang's BMI signature). The compile environment is recorded inside the BMI itself; consume time loads it and compares. A hypothetical 64-bit collision on the single command_hash therefore degrades to a slow re-precompile, never a miscompile. See `_pcm_command_hash` and `_pch_command_hash` docstrings.

**Header-unit `-isystem` immutability contract.** Header-unit cache keys (`_gcc_header_unit_gcm_destination`, the clang-equivalent path) fold in `compiler_identity` and `args.flags.hash_relevant("cxx")` but pass `transitive_content_hash=""` — the resolved header's bytes are NOT in the cache key. That is sound when the header is genuinely immutable across builds (a compiler-shipped system header, a pinned SDK header, a vendored third-party header the user has declared frozen), and broken when the header is project-mutable: gcc's consume-time BMI check is flag-aware, not content-aware, so an edit to a project header reached via header-unit caching would leave the stale `.gcm` in place and import the wrong declarations on the next build. The contract that keeps this safe: **`import <h>;` only resolves to a header unit when the header is reachable through a system-include search flag** (`-isystem`, `-isysroot`, `-iframework`, `-idirafter`, `--sysroot=`). `-I` and `-iquote` are project search paths and are deliberately excluded from `_extract_system_include_path_flags` — a header reached only via `-I` cannot become a header unit, and `import <h>;` against such a header fails the same way as a non-existent header. Routing a header through `-isystem` is therefore the user's opt-in to the "this input does not mutate between builds" contract; edits to that header are intentionally invisible to the cache. If the user actually needs the header to participate in normal mtime-driven rebuilds, they should NOT route it through `-isystem` (and should NOT `import <h>;` it — `#include "h"` keeps the standard content-tracked path).

**Per-compiler details.** Clang writes `.pcm` files directly to the cache path via `--precompile -o <pcm_path>`; importers reference each cached `.pcm` via `-fmodule-file=<name>=<pcm_path>`. The token form (e.g. `<vector>`) is shell-quoted at flag emission to keep `<`/`>` from being interpreted as redirection. GCC reads/writes `.gcm` files via `-fmodule-mapper=<mapper-path>`; the mapper file maps module names (or, for header units, the resolved absolute system-header path obtained from `g++ -M -x c++ -`) to cache paths. The cache key (`command_hash`) folds in: `compiler_identity` (realpath + size + mtime_ns), hash-relevant `cxxflags_tokens`, per-file magic flags, compiler-injected extras (`-stdlib=libc++`, `-Wno-reserved-module-identifier`), source path, transitive content hash, and a `stage` marker (`clang_module_interface` / `clang_header_unit` / `gcc_module_interface` / `gcc_header_unit`).

**Filename escapes.** Module partition separators (`:`) and header-unit path separators (`/`) are escaped to `^^` (`_NAME_ESCAPE` in `build_backend.py`) for the on-disk filename — `_` and `-` would collide with characters that legitimately appear in identifiers and header names. The compiler-facing mapper key keeps the original `:`.

**Transitive header `import M;` / `import <h>;` is supported.** A `.C`/`.cpp` whose only module interaction (a named `import M;` OR a header-unit `import <h>;` / `import "h";`) arrives through `#include "wrap.H"` gets both the compiler's modules-mode flag and a BMI edge into its compile rule — after preprocessing the import is part of the consumer TU, so the consumer compile must resolve it. Both `_compiler_module_flags_for` and `_wire_module_inputs` walk `header_dependencies(filename)` and re-check, so the consumer compile sees `-fmodules-ts` (gcc) / `-fprebuilt-module-path` (clang flat-dir for named modules) and waits on M's producer (avoiding a `-j` race). For **transitive header units specifically**, the symmetry has two extra moving parts beyond the named case: (1) the header-unit precompile pre-pass (`build_graph`, the `all_header_imports` loop) unions each TU's *transitive* header-unit tokens via `_transitive_header_unit_imports`, so the precompile rule, the `_header_unit_artefact` entry, and the gcc mapper entry all exist even when *no* source imports the header unit directly; (2) clang's consume-flag emission in `_compiler_module_flags_for` unions the same transitive token set and emits `-fmodules` + `-fmodule-file=<token>=<pcm>` for it — clang **requires `-fmodules`** to accept a header-unit import even when the BMI is supplied (without it: `error: header file <h> ... cannot be imported because it is not known to be a header unit`). Pre-fix, clang gated those flags purely on the TU's *own* `module_header_imports`, so a transitive-only consumer compiled without them and failed; gcc tolerated the gap only when some *other* source imported the same header unit directly (seeding the mapper). Only *named* imports from transitive headers open the named gate (`import M;`, `import M:P;`); a bare `import :P;` in a non-module header has no resolvable owning module (the header isn't itself a module fragment) and is skipped — that's malformed C++ the compiler would reject anyway. The inline `transitive_named_imports` flag's condition already folds in transitive header-unit imports (so the modules-mode gate opens for them too); the comment there spells this out. Cache-invalidation cost: every TU newly captured by the transitive check gets the modules flag added → `cmd_hash` shifts → one-time cas-objdir miss → steady state restored next build. Regression guards: `test_build_backend.py::TestTransitiveHeaderModuleImports` (11 cases covering gcc/clang named-flag + BMI wiring, clang header-unit consume-flags, the `_transitive_header_unit_imports` collector, no-trigger, partition-skip, and the ordering-filter pass-through for CAS-only mode) and the cross-backend e2e `cxx_modules_transitive_header_unit` example + `test_transitive_header_unit_builds_on_gcc_and_clang` (forces both toolchains explicitly to pin the gcc-passes/clang-fails asymmetry that is the bug's signature).

**Prebuild skip requires the BMI, not just the `.o`.** A gcc named-module interface writes its BMI (`.gcm`) as a *side effect* of producing its `.o`, at the path `-fmodule-mapper` resolves — so the `.o` existing is not proof the artefact is whole. `_prebuild_aux_artefacts` therefore skips a module-interface rule only when BOTH the `.o` and the BMI are present (`_module_interface_bmi_by_output` + the `_module_iface_bmi_path` hook); when the `.o` is cached but the BMI is gone it force-recompiles the interface (`skip_if_exists=False`) to regenerate the side effect (the `.o` rewrite is content-identical via temp+rename). The sharp case is bazel's **"outside" cas-pcmdir layout**: there the BMI lives only in workspace-local `.ct-bazel-pcm/` (bazel hermeticity forces workspace-local BMIs; only objects stay CAS-shared), so a rebuild with a warm shared cas-objdir but a wiped `.ct-bazel-pcm/` (e.g. `git clean`, fresh checkout sharing an external object cache) would otherwise skip on the `.o` alone and leave bazel failing on the missing declared BMI input. `BazelBackend._module_iface_bmi_path` overrides the base (which checks the cas-pcmdir path) to resolve the mapper location — cas-pcmdir for "inside", `.ct-bazel-pcm/` for "outside". No-op for clang (its `.pcm` is a rule *output*, already existence-checked). Regression guard: `test_examples_end_to_end_cross_backend.py::test_bazel_outside_layout_regenerates_bmi_after_staging_wipe` (build → wipe `.ct-bazel-pcm/` → rebuild must regenerate the BMI; fails without the fix).

## Variant resolution: load-bearing pieces

- **`_DEFAULT_VARIANT_CANONICAL_ORDER`** (tuple in `configutils.py`) is the SINGLE source of truth for canonical token ordering. Bundled `ct.conf` does NOT redeclare it as data; it shows the full list as a commented-out copy-paste starting point. `test_bundled_ct_conf_comment_example_matches_builtin` is the drift guard.
- **Override hierarchy** for canonical order: `--variant-canonical-order=<tokens>` CLI > `CT_VARIANT_CANONICAL_ORDER` env > `variant-canonical-order = ...` in any ct.conf level > builtin. Implemented in `get_canonical_order(argv=...)`, which scans argv early since the order is needed at create_parser time, before configargparse parses.
- **`extends = ...`** in any axis or composite conf names parents. Resolution is DFS with cycle detection (`VariantResolutionError`), first-visit diamond dedup, and "highest-priority conf wins" for which `extends` value is taken when multiple files share a name. The traversal order of `extends` is behavior-affecting — `extends = werror, gcc` produces different flag layering than `extends = gcc, werror`. Both `test_bundled_composite_extends_obeys_canonical_order` (CI) and `_check_extends_canonical_order` (runtime) enforce this.
- **Composite override** (`<canonical_name>.conf`) tunes a composition. Default semantics: layers on top of the canonical-token atoms. Explicit `extends = ...` in the composite replaces that default. Multiple composites in the hierarchy ALL contribute to `flat_paths`; the highest-priority composite's `extends` directive steers chain-seed selection.
- **Parse cache** (`_parse_conf_file_cached`, `@cache` by absolute path) collapses ~13 file opens per parseargs flow to one open per file per process. `test_resolve_variant_parses_each_conf_at_most_once` is the regression guard. `clear_cache()` resets per-test.
- **`append-*` / `prepend-*` accumulate across the hierarchy AND across CLI** via `apptools._ComposingArgumentParser` + `_AccumulatingConfigFileParser`. Stock configargparse's `already_on_command_line` check discards `action='append'` keys for the same option strings that any higher-priority source (conf-file or CLI) already injected — so without this override, `--variant=gcc,release,extras` would keep only `extras.conf`'s `append-CXXFLAGS` value, and any CLI `--append-CXXFLAGS=...` would wipe out the entire conf-file contribution. The override has two halves: (1) `_open_config_files` concatenates the whole conf hierarchy into one stream, and the accumulating parser merges duplicate `append-*`/`prepend-*` keys into a list; (2) `parse_known_args` strips CLI `--append-*` / `--prepend-*` tokens before super-call so they don't suppress conf-file injection, then re-appends them to the parsed namespace so CLI still applies last (compiler honors last occurrence of conflicting flags = CLI override semantics preserved). Regression: `TestAppendFlagsAccumulateAcrossConfHierarchy` in `test_apptools.py`.
- **Opinionated bundles** (`dev`, `ci`, `production`, `safety`, `perf`, `secure`) are tiny `<name>.conf` files using `extends = ...` to compose curated axis sets.
- **Legacy `variantaliases = {dict}`** is hard-failed by `_check_legacy_variant_config_keys` (byte-level scan, line-anchored). See `README.ct-config.rst` "Upgrading from variantaliases".

The flow inside one parseargs invocation: `create_parser` → `extract_variant(argv)` → `resolve_variant(variant, argv)` (consults builtin/conf/env/CLI for canonical_order, resolves axes, returns `VariantResolution`). After configargparse parses, `_commonsubstitutions` re-runs `canonicalize_variant_input(args.variant, argv=args._argv)` and a second `resolve_variant(argv=args._argv)` to populate `args._variant_resolution` for the `-vv` provenance trace. The `_argv` stash on `args` is what lets the second resolve hit the `--config=path` short-circuit branch.

## `${CONF_DIR}` placeholder + conf-file provenance

`${CONF_DIR}` expands at conf-parse time inside `_AccumulatingConfigFileParser.parse()` to the absolute directory of the conf file the value originated from — generic across every key, not just `*-PATH`. Opt-in: bare relatives still pass through verbatim, so the placeholder is the explicit way to anchor a value to its conf file. Closes the 2026-05-15 upstream wrong-flavor bug, where bare-relative `prepend-PKG-CONFIG-PATH` resolved against the consumer's cwd (`cas-pchdir/<variant>/<hash>/`) at compile time instead of the conf-file directory.

Multi-conf concatenation: `_open_config_files` prepends a `# --- <name> ---` + `# __ct_conf_dir__=<abs/dir>` header pair before each file's body. The parser tracks a `marker_expected` flag flipped on by the segment header and consumed by the next non-blank line — so a user comment that happens to contain the marker string mid-file can't poison subsequent expansions.

Provenance side channel: `(expanded_value, source_file, lineno)` per parsed entry, accumulated in `self._provenance` and exposed via `_ComposingArgumentParser.get_conf_file_provenance()` (returns a shallow copy; the inner tuples are immutable). `_setup_pkg_config_overrides_locked` consumes it at `verbose >= 4` to attribute every emitted `Prepended/Appended pkg-config path: ...` with `(from /abs/conf:N)` / `(from CLI)` / `(auto-discovered: cwd|gitroot)`. The `args.*` shapes are unchanged — provenance is a parallel channel, not a wrapping of the parsed values.

## `PKG_CONFIG_PATH` ordering: conf-layer priority

`_AccumulatingConfigFileParser` hands `prepend-PKG-CONFIG-PATH` / `append-PKG-CONFIG-PATH` to `_setup_pkg_config_overrides_locked` ordered `[low-priority conf, ..., high-priority conf, CLI]` — the same accumulator order every `prepend-*` / `append-*` key uses. Compiler-flag slots emit left-to-right and rely on the compiler's "last token wins" rule, but `PKG_CONFIG_PATH` resolves leftmost-first, so the function reverses both lists before joining with `:` so CLI > high-priority conf > low-priority conf survives the inversion. Symmetric for prepend and append; within each group the highest-priority source ends up leftmost (winning). Regression: a project-`ct.conf` `prepend-PKG-CONFIG-PATH` previously shadowed the same key set by a higher-priority axis conf, breaking per-axis ABI-pinned `.pc` selection.

## Workspace-relative compile paths: full Round 3 history

`_inject_ffile_prefix_map` skipped per-slot when the user already supplied any `-f{file,debug,macro,canon}-prefix-map=`. Linker symmetry via `canonicalize_for_command` / `canonicalize_path_for_command` (target substituted, not `<GITROOT>` sentinel); `_create_link_rule` / `_create_shared_library_rule` route LDFLAGS through them so RPATHs and `--version-script` paths under the gitroot are target-prefixed in the argv. `canonicalize_for_cache_key` also rewrites the OLD side of those flag families, otherwise two users' auto-injected tokens would re-leak the gitroot through the very flag meant to scrub it.

PCH/BMI byte-identity is empirically unachievable: gcc's PCH (`.gch`) and BMI (`.pcm` / `.gcm`) on-disk layouts encode pointer offsets that depend on the build cwd path *length* — same-length cwds yield identical bytes; differing lengths shift internal offsets. No combination of `-ffile-prefix-map`, `-fdebug-compilation-dir`, `PWD=/proc/self/cwd`, or symlink-chdir tricks fixes it. **Downstream `.o` byte-identity IS achievable** via the precompile rule emitters (`_create_pch_rules`, `_create_clang_module_interface_rules`): (1) source path passed *relative to* `self._anchor_root`, gated on `_is_under(source, anchor_root)`; (2) `BuildRule.cwd = self._anchor_root`. Backends honour `cwd` by `cd <cwd> && ` inside the lock wrapper (make/ninja via `_wrap_compile_cmd(..., cwd=)`) or `subprocess.run(cwd=)` (shake/slurm via `execute_compile_rule(..., cwd=)` → `atomic_compile(..., cwd=)` → `_run_with_signal_forwarding(..., cwd=)`). Without this discipline, alice's PCH bakes `/home/alice/proj/...` into its DWARF `.debug_line_str` include-dir table and any consumer (including a peer reading from a shared cas-pchdir) inherits alice's paths into their `.o`.

For gcc C++20 modules, the per-build module-mapper file at `<bindir>/.module-mapper.txt` (outside the gitroot, so `-ffile-prefix-map` doesn't reach it) leaks its absolute path into `DW_AT_producer`'s recorded compile-flag string. `_compiler_module_flags_for` appends `-gno-record-gcc-switches` whenever `-fmodule-mapper=` is present — DW_AT_producer keeps the gcc version + target triple but drops the per-user mapper path.

Acceptance: `test_two_checkouts_produce_byte_identical_o_with_shared_cas_p{ch,cm}dir` (alice and bob share a single cas-pchdir/pcmdir, assert their `.o` outputs match — what NFS-shared CAS users actually need). The companion `test_two_checkouts_produce_byte_identical_cas_p{ch,cm}dir` tests are kept `xfail(strict=False)` as **negative controls** that pin the empirical `.gch`/`.pcm` divergence — if a future change accidentally achieves byte-identity, the marker can be dropped.

### Relative `--cas-*dir` are gitroot-anchored (was a bug)

`resolve_cas_directory_arguments` anchors a *relative* cas dir to the gitroot — `wrappedos.normpath(os.path.join(find_git_root(), value))`, reusing the resolver's existing `git_root` (the same string `self._anchor_root` uses). `os.path.join` passes absolute values through unchanged, so default/absolute cas dirs are untouched; only relative ones are anchored. A relative `--cas-*dir` therefore means "relative to the gitroot" (matching the gitroot-anchored default). The anchoring is gated on `wrappedos.realpath(git_root) != os.path.realpath(os.getcwd())` (the cached wrapper for the repeated `git_root`, a one-off direct read for the live cwd) — applied only when invoked from a *subdir* of the gitroot (the bug condition). `find_git_root()` falls back to returning the cwd outside a repo, so this gate also leaves bare-relative cas dirs literal outside a repo (the no-auto-anchor contract; `test_conf_env_expansion`).

Why this is load-bearing (the bug it fixes): the precompile rules deliberately run under `cwd = anchor_root` (`cd <gitroot> && g++ ... -o <cas-path>`) with a gitroot-relative *source* for cross-user PCH byte-identity (`_create_pch_rules`, `build_backend.py:1072-1084`; the `pcm_rule_cwd = anchor_root` branch at `:2621-2626`), but emit the `-o` output **as given**. Before anchoring, a relative `--cas-pchdir` invoked from a *subdir* of the gitroot produced a relative `-o` that `make` created relative to its own cwd while the wrapped `cd <gitroot> && g++ -o <relpath>` resolved against the gitroot — a different, nonexistent dir — so gcc died with `cannot create precompiled header ...: No such file or directory`. Anchoring at the resolver guarantees the rule always gets an absolute `-o`, fixing PCH and PCM uniformly across all backends, before any backend-specific rule generation runs.

**Gitroot-anchoring, not cwd-anchoring (`os.path.abspath`)** is the correct form: `canonicalize_path_for_cache_key` is a textual string-prefix op, so cross-user byte-identity needs the cas-dir string to share the exact `anchor_root` prefix — gitroot-anchoring guarantees it by construction, cwd-abspath only does when `getcwd()` and `git rev-parse --show-toplevel` agree textually (they don't under symlinked/NFS-automounted checkouts). Regression guard: `test_relative_cas_dir_bug.py` (relative-from-subdir-anchors-to-gitroot + absolute-under-gitroot + absolute-outside-gitroot/shared-CAS + a compiler-free resolver unit test) over `examples-features/relative_cas_dir_bug/`.

## Magic detection: `MacroState` cache-key details

`MacroState` in `preprocessing_cache.py` separates static compiler built-ins (~388 macros, immutable) from dynamic file `#define`s (variable). Cache keys only hash variable macros — ~80% reduction in key computation cost. Lazy frozenset computation with incremental updates for pure additions (the common case). The build-context portion of the hash also folds in `compiler_identity(args.CXX, anchor_root=...)` (a `realpath|size|mtime_ns` triple resolved via `apptools.compiler_identity`, exposed on `args.flags.compiler_identity`) so an in-place toolchain swap that leaves `args.CXX` unchanged still invalidates per-TU object cache entries (symmetric with the PCH cache key). The realpath segment is routed through `canonicalize_path_for_cache_key` so an in-workspace wrapper script (coverage shim, sccache/distcc wrapper) at e.g. `<gitroot>/tools/cxx-wrapper.sh` doesn't leak its per-checkout prefix into the cache key. The build-context hash also folds a snapshot of the CPATH-family env vars (`INCLUDE_PATH_ENV_VARS`: `CPATH`, `C_INCLUDE_PATH`, `CPLUS_INCLUDE_PATH`, `OBJC_INCLUDE_PATH`; unset ≡ empty, matching `_link_environment_snapshot`) because they change which headers the compiler resolves without appearing in any flag string — ccache hashes the same four. `DirectHeaderDeps` symmetrically appends those directories to its flat include search list (after CPPFLAGS-derived `-I` dirs, CPATH first, mirroring gcc's `-I`-then-`-isystem` staging) so env-only headers are walked and content-hashed into `dep_hash`; the link-side `_LINK_ENVIRONMENT_VARS` list in `backend_command_args.py` is deliberately separate and must stay byte-stable.

## Naming history

Prior to a rename, the object CAS was called `shared-objdir/` (default `{git_root}/shared-objdir/{variant}/`) and the PCH CAS was `shared-pchdir/`, with CLI flags `--objdir` / `--pchdir`. The "shared object" overload conflicted with Linux `.so`. **No backward-compat alias**: existing `ct.conf` files setting `objdir` / `pchdir` must rename those keys to `cas-objdir` / `cas-pchdir`, and on-disk `shared-objdir/` directories from earlier builds are no longer consulted (safe to delete).

## Shared / multi-user cache: object-trim currency semantics

`ct-trim-cache`'s object-cache currency check (`build_current_hash_set`) reads the **invoking checkout's** git HEAD — it builds the set of 12-char file-hash prefixes for that checkout's tracked files and discards any cached object whose `file_hash` field isn't in the set. On a shared pool used by multiple branches or users this means objects that are current for another checkout look non-current here, and a run with the default settings (no `--max-age`, `--keep-count 1`) can evict them down to one per basename.

- **Preferred control on shared pools**: `--max-age=N`. This limits removal to entries older than N days regardless of which checkout considers them current, so cross-branch objects are preserved as long as they are being actively rebuilt.
- **Wrong-checkout signal**: if a non-empty scan produces zero current objects and `--max-age` was not supplied, `warn_if_wrong_checkout` emits a stderr warning (at `verbose >= 0`, on network filesystems only — local-disk pools are single-user by convention). The guard uses `should_parallelize_scan` as the network-FS proxy, consistent with the parallel scan's own FS classification.
- **Correctness note**: the 12-char hash prefix is deliberately loose (collision over-retains, never over-deletes), and the currency set is built fresh per invocation from the live git index — there is no cross-invocation state to invalidate.
- **Size bounding is separate from currency**: `--keep-count`/`--max-age` bound only the *non-current tail*; current objects are kept unconditionally, so they set an irreducible per-checkout floor. The explicit `--max-size` budget (`enforce_budget`) is the only knob that caps a pool by bytes — it evicts non-protected entries oldest-first (peer-safe: never current objects or `nlink>1` published artefacts; an unmet budget is reported, not forced) and may go below `--keep-count` for rebuildable entries. It is **per-pool**, not aggregate, so eviction is predictable regardless of cache-processing order. Orphaned-variant cells (`--purge-unresolvable`) and orphaned producer temps (`reclaim_orphan_temps`, age-floored) are the two other byte sources the per-bucket trim alone never reclaims.

## Test-rerun skip semantics (CAS-only mode)

`make runtests` and the Python-side `_run_tests` filter decide whether to re-execute a test by consulting the `<cas-path>.result` sidecar, NOT `mtime(<bin>/<test>.result) >= mtime(<bin>/<test>)`. The mtime check is wrong here because the published `bin/<variant>/<test>` is a hard-link of the cas-exedir entry — it inherits the cached entry's original creation time, so any cas-exedir hit produces a stale-looking exe relative to a newer marker file. With the content-keyed sidecar, the rule becomes `<cas-path>.result: | bin/<variant>/<test>` (order-only on the published exe so make builds it first; no normal prereqs so existence of the marker is sufficient). User-visible upshot: two workspaces sharing a `--cas-exedir` also share test-success markers — workspace B's first `runtests` correctly skips a test workspace A already ran for the same exe content. Tests with `$PWD`-dependent observable side effects should set `--use-mtime=True` to opt out (the marker assertion is "this exe's bytes once exited 0," not "this exe ran in this workspace"). Legacy pre-fix `<bin>/<test>.result` files left over from earlier installs are harmless dead bytes.

## Argparse two-stage flow + flag string drift

After `apptools.parseargs` returns, `args.flags` is a frozen `Flags` dataclass. Convenience methods: `hash_relevant(slot)` returns the slot's tokens with `-D`/`-U` and diagnostic-only flags removed (used by cache-key hashing); `existing_include_paths(slot)` and `append_include(path, slots=...)` form the include-path dedup API used by `apptools._add_include_paths_to_flags`. Because `Flags` is frozen and tuple-backed, instances are hashable and consumers cannot mutate the underlying tokens — `append_include` returns a new `Flags` via `dataclasses.replace`.

**Mutation invariant.** `args.{CPPFLAGS,CFLAGS,CXXFLAGS,LDFLAGS}` raw strings, `args.{*}_tokens` lists, and `args.flags` are populated once at the end of `parseargs` and must not be mutated afterwards — `args.flags` would silently drift from the raw strings. All known mutation sites (`substitutions`, `_add_include_paths_to_flags`, project version macros, pkg-config, CPP/CXX unification) run BEFORE that point. `apptools.check_flag_string_drift(args)` compares the current raw flag strings against the snapshot recorded at parseargs end (`args._flag_string_snapshot`) and raises `RuntimeError` naming the offending slot if anything has changed.

## `//#GIT=` external fetch (`fetch.py` + `cake._fetch_and_register_externals`)

`//#GIT=<url>[@<ref>]` folds an external git repo into the build. On a `ct-cake --auto` build the settled targets are scanned to a fixpoint; each declared external is cloned/pulled and its root (plus `include/`) is registered on the include path so the main repo can `#include`/link it. `ct-fetch` is the standalone entry point over the same machinery (clone / `--update` / `--no-fetch` / `--status`).

**Why widen `args.INCLUDE` + re-run `substitutions()`, not touch flag slots.** After `parseargs` the flag slots (`args.{CPPFLAGS,…}`, `args.{*}_tokens`, `args.flags`) are frozen (see the mutation invariant above); mutating them post-parse trips `check_flag_string_drift`. `_fetch_and_register_externals` instead appends the discovered external dirs to `args.INCLUDE` — NOT a frozen slot — and returns `True`, which makes cake re-run `apptools.substitutions()` to redistribute INCLUDE into the *FLAGS and re-finalize the frozen `args.flags`. That is the sanctioned path for widening the include set after parse: go through INCLUDE + substitutions, never edit a flag slot directly.

**Why `_augmented_headerdeps` no longer deep-copies args (and `BuildContext` has no `__deepcopy__`).** The header walk must reach INTO already-fetched externals to discover their transitive `#include` graph and further `//#GIT=` declarations. It does this by passing the external include dirs through `headerdeps.create(..., extra_include_dirs=[...])` — a keyword-only parameter threaded through `HeaderDepsBase.__init__` onto `self._extra_include_dirs`. `DirectHeaderDeps._initialize_includes_and_macros` appends them (as raw path strings, so paths with spaces survive without a shlex round-trip) to the include search list it builds from `CPPFLAGS`; `CppHeaderDeps` appends them as `-I` tokens to the `cpp -MM` command line. The caller's real `args` (and its frozen `args.flags`) is neither mutated nor copied. Historically this was done by `copy.deepcopy(args)` with `-I` flags appended to the copy's `CPPFLAGS`; but `args` transitively references the live `BuildContext` via `args._context`, and the context owns a `threading.Lock` and `stringzilla.Str` caches that are not deep-copyable, which forced a `BuildContext.__deepcopy__` returning `self` — a global identity-aliasing hook on a shared type, a latent trap. That deepcopy was the ONLY production caller, so both it and `__deepcopy__` are gone. The `context.analyzer_args` save/restore try/finally in the fixpoint driver is STILL required: `HeaderDepsBase.__init__` calls `set_analyzer_args(args, context)`, which now writes the *real* args into `context.analyzer_args`; the `finally` restores whatever the caller held before. The analyzer/preprocessing caches key on the content-hash of include *names*, not resolved paths, so sharing them across rounds cannot cross-contaminate.

**Bounded fixpoint + per-round cache clear.** `_MAX_FIXPOINT_ROUNDS = 50` caps the loop: a correct run adds at least one new external name per round or stops, so a real graph converges in a handful of rounds; the cap turns a hypothetical non-converging cycle into a named `FetchError` rather than an infinite loop. At the end of every round `wrappedos.clear_cache()` drops the globally-`@functools.cache`'d stat answers — a "file missing" answer cached while an external header did not yet exist on disk would otherwise stick and blind the NEXT round's `_find_include` to the freshly-cloned sources, breaking transitive discovery (external A's headers declaring `//#GIT=` for external B).

**Transport-protocol restriction + `//#GIT_ALLOW_PROTOCOL`.** `_git_env` sets
`GIT_ALLOW_PROTOCOL=file:git:ssh:http:https` via `setdefault` (ambient env
wins). `ext::` is excluded so an untrusted `//#GIT=ext::<cmd>` cannot execute a
command at fetch time. A project widens the set with a `//#GIT_ALLOW_PROTOCOL=`
magic comment (colon-separated git protocol list); `extract_git_allow_protocols`
gathers them, `fetch_externals` unions the tokens per round, and the resulting
string is threaded to `_run_git` via the `_git_run_ctx` ContextVar
(`_GitRunContext.allow_protocol`).

**Parallel-fetch signal safety.** Worker-thread git clones register their child
pgid in a thread-safe `_LiveChildren` set (via
`_run_with_signal_forwarding`'s `on_child_start`/`on_child_end` hooks, carried
on the `_git_run_ctx` ContextVar). `_resolve_new` installs a MAIN-thread
`graceful_shutdown` handler that forwards SIGINT/SIGTERM to every registered
child and raises `KeyboardInterrupt`; the result loop catches only `Exception`
so interrupts propagate, and `executor.shutdown(cancel_futures=True)` cancels
queued clones on the first error or on interrupt. Without this the in-worker
forwarder inside `_run_with_signal_forwarding` (a no-op off the main thread)
would let `start_new_session=True` git children orphan on Ctrl-C / SIGTERM.

**Present-checkout origin check.** `_handle_present` compares the on-disk
`origin` (`_origin_url`, normalized by `_normalize_remote_url`) against the
declared `ext.url` and WARNS on mismatch — the sibling-dir default layout can
otherwise silently share `../<name>` between two projects declaring the same
name from different URLs.

**Linked-worktree guard (A23).** The sibling-dir default is also exactly where
users keep their own `git worktree` checkouts, so a `//#GIT=` name collision
can land the managed location on a linked worktree of a repo the user is
actively working in. `_is_linked_git_worktree` (git-dir ≠ git-common-dir)
detects it, and `_handle_present` refuses ALL mutating git ops there: a
differing ref, or any `--update` (even at the right commit — a branch
fetch/fast-forward writes refs into the user's MAIN repo via the shared git
dir), raises a named `FetchError` pointing at `--git-path`/`--externals-dir`.
A worktree already satisfying the declaration (unpinned, or detached at the
requested SHA/tag) is used as-is. Distinct from the A19 nested-plain-dir
hijack guard in `_is_git_work_tree`: a linked worktree IS a work-tree
toplevel, so it passes A19 and needs its own check.

**One shared `_fixpoint_scan` driver, parameterized by three callbacks.** `fetch_externals` (used by cake and `ct-fetch`'s clone/update path) and `gather_external_status` (`ct-fetch --status`) both delegate the round loop to `_fixpoint_scan(target_files, args, context, *, externals_dir, root_selector, scan_round, on_not_converged)`. The driver owns the shared skeleton: the bounded `for _round in range(_MAX_FIXPOINT_ROUNDS)`, building `_augmented_headerdeps` over the roots `root_selector()` returns, computing `_reachable_sources`, breaking when `scan_round` reports no new work, `wrappedos.clear_cache()` between rounds, the `else:` non-convergence raise, and the `context.analyzer_args` save/restore try/finally. The three callbacks (closures over each caller's own accumulator dicts) carry the ONLY behaviour that differs:

- **`root_selector()`** — which already-known roots to widen the search into. fetch returns every resolved/cloned root (`[r.path for r in resolved.values()]`); status returns only roots ALREADY present on disk, since it never fetches to expand the graph.
- **`scan_round(reachable)`** — record newly-declared externals (dedup by name) and take the per-mode action. fetch RAISES `FetchError` on a conflicting URL/ref (and a case-collision) and resolves the round's new externals; status only WARNS on a conflict (`--status` is a report, first-wins) and records. Returns `True` while a round found new work, `False` at convergence.
- **`on_not_converged()`** — the mode-specific runaway-loop `FetchError`.

This replaces the former two hand-synced near-duplicate loops (previously cross-linked by "keep in sync, do NOT extract a shared generator" NOTE comments); the load-bearing differences now live entirely in the three callbacks, so there is one skeleton to maintain.

**Fetch resolves a round's new externals in parallel.** The externals newly discovered in one round are independent (unique names, per-target `FileLock` sidecars, separate git endpoints, each writing only its own `resolved[name]`), so fetch's `scan_round` resolves them in a bounded `ThreadPoolExecutor` (`_MAX_FETCH_WORKERS` cap, effective workers `min(len(new), cap)`; a single external takes the sequential path). Each worker acquires its own `<target>.lock` sidecar. Determinism is preserved: results are assigned into `resolved` in declaration order, and on failure the FIRST error in declaration order is re-raised (every future is awaited, so no exception is swallowed) — the same `FetchError` a sequential run would have surfaced.

**Status computes each external's on-disk status once.** `gather_external_status` never mutates the filesystem, so a declared external's `ExternalStatus` cannot change mid-run. It is cached by name (`status_cache` + `_status_cached`) and reused across the per-round `root_selector` calls and the final result pass, instead of re-running `_status_for`'s three git subprocesses `N × rounds` times.

**Externals live outside the gitroot.** The default externals dir is the gitroot's *parent* (`resolve_externals_dir`): each external lands at `../<name>`. So external artifacts are legitimately not cross-workspace-cache-identical — the "no cross-worktree caching" design applies here too, and that's by intent, not a gap.

## Wild linker: the one compiler-rewritten linker axis

Linker axis confs are otherwise purely declarative (`gold.conf` =
`-fuse-ld=gold`, etc.), but wild has two documented exceptions in
`apptools`:

- **`wild` axis** — `wild.conf` emits the canonical token `-fuse-ld=wild`
  (what gcc ≥ 16.1 wants). `_normalize_wild_linker` (run in
  `_commonsubstitutions`, before the `Flags` freeze) rewrites it to
  `--ld-path=wild` when the effective link driver (`_effective_link_driver`:
  `args.LD` else `args.CXX`) is clang — clang rejects `-fuse-ld=wild`
  ("invalid linker name") without an `ld.wild` symlink, and wild's installer
  doesn't make one. The bare name `wild` (not an abs path) keeps per-user
  paths out of the link key.
- **`wild-B` axis** — a comment-only conf (like `ld.conf`). Selection is
  detected via the variant token (`_variant_has_axis`), and
  `_materialize_wild_b_searchdir` creates `<gitroot>/.ct-wild-ld/ld → wild`
  and injects `-B<dir>`. Gitroot-anchored so `canonicalize_for_cache_key`
  neutralises it in the link key. Works on any gcc (no `-fuse-ld=` support
  needed).

`_check_wild_linker_usable` (next to `_check_resolved_compiler_available`)
fails fast: wild missing → install hint; `wild` axis + gcc < 16 → use clang /
upgrade / `wild-B`. The bazel backend recognises `--ld-path=` (so the clang
rewrite doesn't get clobbered by its `-fuse-ld=gold` default); **`wild-B`
under bazel is unsupported** — its `-B` selection signal isn't a recognised
linker flag there.

## Startup guards

Three startup-time guards in `apptools` catch common misconfigurations early: (a) `_check_resolved_compiler_available` — toolchain axis pinned a compiler that isn't on PATH; (b) `_check_wild_linker_usable` — the wild linker is selected (`-fuse-ld=wild` / `--ld-path=wild`, or the `wild-B` axis) but the `wild` binary is missing, or the `wild` axis is paired with gcc < 16 (which can't drive `-fuse-ld=wild`); (c) `_check_compiler_supports_requested_standard` — `-std=c++NN` exceeds the resolved compiler's major version per `_STD_MIN_COMPILER_VERSION`. All name the variant chain in the diagnostic so the user knows which axis to swap.

## `wrappedos` preference and the documented skip cases

Prefer `compiletools.wrappedos.<fn>` over `os.path.<fn>` whenever the wrapper exists. The wrapped surface is `realpath`, `dirname`, `basename`, `getmtime`, `isfile`, `isdir`, `isabs`, `getsize`, `join`, `normpath` (plus `_sz` StringZilla variants for the same set minus `normpath`). All are `@functools.cache`'d on the input string; on hot paths that re-stat the same source/header thousands of times (dep walks, cache-key canonicalisation, object-file deduping) this turns the repeat work into a dict lookup. `join` and `normpath` are pure string ops where the wrapper exists mainly for API uniformity, but the cache is still free and removes a "well it's just normpath, who cares" temptation footgun.

Skip the wrapper — and use `os.path.<fn>` directly — only when caching would be **wrong**. Three patterns recur:

1. **Stats of lock/sidecar files whose existence or mtime can change concurrently** with the call. `locking.py:434` and `lock_utils.py:34` have inline `CRITICAL` comments explaining why; replicate that pattern (comment + uncached call) anywhere a lock file or sidecar is the target of the stat.
2. **Build-output existence checks** that happen AFTER the rule that produces the output has run. Examples: `build_backend.clean()/realclean()`, `trace_backend._make_trace_entry` (post-execute output check), `ninja_backend` (post-build log read), `trim_cache` / `cache_report` (post-build cache directory walks). The cached "missing" answer from a pre-build call would be stale once the build finished.
3. **Relative-path inputs to a cached fs-query, subject to chdir.** Every cached `wrappedos` fs-query keys on the *input string*, so feeding it a **relative** path lets the first call lock in that path's resolution against the then-current cwd; any later `chdir` makes the cached answer stale. This applies to the whole stat-like surface — `realpath`, `isfile`, `isdir`, `getmtime`, `getsize` — not just `realpath`. The cardinal example is `bazel_backend.py`'s `_default_base_dir`: the **relative** input `BUILD.bazel` resolved once would return the wrong absolute path after any later `chdir` (production doesn't chdir mid-build, but tests do), so it uses `os.path.realpath` direct. **Rule: pass absolute paths to cached fs-queries whenever cwd can change between calls; use the direct `os.path.<fn>` only for a *relative* input whose resolution depends on cwd.** The operative test is whether the **input string** is absolute — a value that merely *originates* from the cwd is fine to cache if it's already an absolute string (e.g. `os.path.realpath(os.getcwd())` or `realpath(find_git_root())`: `getcwd()`/`find_git_root()` return absolute strings, so the chdir footgun cannot occur; whether to cache those is a hot-vs-cold-path call, not a correctness one).

Functions NOT in the wrappedos surface — `os.path.{expandvars, expanduser, exists, commonpath}`, `os.environ`, `os.getcwd`, `os.walk`, `os.listdir`, `os.makedirs`, `os.symlink`, `os.unlink`, `os.replace`, `os.readlink`, `os.getpid`, `os.access`, `os.X_OK` — MUST be called as `os.<fn>` directly. Adding a wrapper is a one-line `@functools.cache` decoration in `wrappedos.py` plus a `clear_cache()` registration; do it when a function lands in a hot path with stable inputs (the bar that `normpath` cleared in commit `ed405353`).
